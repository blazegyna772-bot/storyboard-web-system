# 决策记录

## D-001 当前阶段定位

当前阶段定位为：需求方案 + 设计方案 + 可扩展前端原型。

不是生产系统。

## D-002 基础工作台壳子

采用：顶部窄栏 + 左侧可收起导航 + 中间主工作区 + 右侧覆盖式工具抽屉 + 底部执行反馈。

左侧只做一级模块导航，不承载全文剧本、项目总览或重复流程信息。

顶部放项目、保存、生成、导出和工具入口。

右侧工具抽屉默认收起，打开后覆盖页面，不挤压主工作区。

## D-003 废弃：独立上下文包数据层

旧方案曾将“上下文包/辅助信息审阅”作为独立数据层和一级页面。

该方案已废弃，不再作为后续开发依据。

替代方案见 D-015：

- 剧本统筹页承载全剧统筹和章节概要。
- 分镜统筹页承载 单集概要/场次概要/分镜设计。
- 视频生成页承载 06。
- 原辅助信息职责进入对应节点产物，不再单独成页。

## D-004 Pipeline 必须现在设计

由于存在 50000 字长文本、多题材、多规则包、多输出格式，系统必须现在设计 Pipeline 骨架。

后期再补会导致重写。

## D-005 当前规则生成只是占位执行器

当前 `scriptQuality`、`storyboard`、`contextPack` 都是规则版占位。

未来应替换或并行接入 LLM executor。

## D-006 Pipeline 骨架先于模型接入

先实现 PipelineStage、RulePack、GenreProfile、DirectorProfile、OutputAdapter 的前端配置与默认骨架，再接入 LLM。

原因：长剧本、多题材、多输出格式会持续扩展，模型接入不能写死在页面组件中。

## D-007 最终产物导向

最终产物只有资产描述/生图提示词、分镜统筹/视频提示词。

所有中间产物必须有明确用途和后续引用。无效、重复、无法被执行阶段使用的信息不进入管线。

## D-008 场级首次规划，旧块级节点暂不采用

首次分镜统筹以一场戏为调用边界，由 分镜设计负责场内空间时序和镜头连续性。

旧方案中的“块级分镜设计”节点暂不采用，避免 分镜工作流 主流程被拆碎。

当前默认重跑粒度：

- 单集概要：影响本集任务、情绪、钩子、承上启下时重跑。
- 场次概要：影响人物进入状态、潜台词、空间关系、道具状态时重跑。
- 分镜设计：影响镜头语言、画面动作、转场和镜头连续性时重跑。

以后如确实需要镜头小块重跑，只能作为分镜设计内部能力，不新增独立节点。

## D-009 人物空间站位归属

人物站位不交给单集概要发明。

如果剧本中有明确进出场、移动、桌边关系、人物相对位置，分镜设计应在整场范围内推导并输出 `scene_spatial_timeline`。

视频提示词只引用已确认分镜和空间时序，不逐镜随意发明站位。

## D-010 中间产物最小有效原则

中间产物必须满足三条：

- 下游明确使用
- 来源可追踪
- 不重复当前执行单元已经能看到的信息

不满足这些条件的信息不进入 Artifact Store。

## D-011 执行反馈形态

底部执行反馈默认是一条窄状态栏，服务普通使用。

展开后是开发/排查大面板，服务 Pipeline、LLM、组件函数和人工修改日志追踪。

## D-012 本地项目先行

P0 阶段先实现 localStorage 本地项目、多项目和版本摘要。

后端、权限、完整 diff、回滚进入后续生产系统阶段。

## D-013 后端必须拆成可插拔车架

当前系统不能把本地文件、LLM 代理、图像代理和项目管理塞在前端临时代码中。

下一阶段要拆出独立本地后端。当前已采用 FastAPI 后端，前端仍是操作台，后端负责：

- 项目根目录配置
- 项目扫描 / 创建 / 删除 / 保存
- 项目文件读写
- LLM/API 代理
- 管线调度
- 运行日志
- Artifact 持久化

生产系统目标不是写死一条流程，而是可插拔生产车架。

后端必须按以下边界设计：

- `core`：项目存储、管线引擎、插件注册、配置加载、日志、Artifact Store。
- `adapters`：LLM、图像、导出、第三方 API 适配器。
- `pipelines`：剧本校检、分集、资产、分镜、提示词等节点。
- `rulepacks`：规则、Prompt、Schema、API 适配器配置。
- `projects`：项目文件夹读写和项目配置。

所有新能力默认按可替换节点处理：

- API 不写死。
- Prompt 不写死。
- Schema 不写死。
- 输出格式不写死。
- Pipeline 节点不直接耦合页面。

换第三方 API 的目标形态是：加载另一套 adapter/rulepack 配置，而不是改业务代码。

## D-014 生产形态

当前建议路径：

1. 先拆独立本地后端 + React 前端。
2. 业务流程稳定后，再用 Electron 打包成桌面应用。

原因：

- 当前需求仍在快速变化，先上 Electron/Tauri 会增加开发摩擦。
- 本地后端承载文件系统、LLM、生图、任务队列、日志和批处理。
- Electron 作为最终交付壳，负责双击打开、自动启动本地后端、隐藏命令行复杂度。

## D-015 取消辅助信息一级页，拆分剧本统筹/分镜统筹/视频生成

独立“辅助信息”一级页面取消。

原因：

- 新的 分镜工作流 LLM 分层工作流已经覆盖原辅助信息职责。
- 集级辅助进入单集概要。
- 场级连续性进入场次概要。
- 资产锁定项进入资产审阅。
- 镜头级连续性进入分镜设计和视频提示词。

一级页面调整为：

1. 项目管理
2. 剧本校验
3. 剧本统筹
4. 资产审阅
5. 分镜统筹
6. 视频生成
7. 任务记录
8. 流程配置

页面边界：

- 剧本统筹：只做全剧统筹和章节概要，产物服务资产审阅和分镜统筹。
- 资产审阅：统一角色、场景、道具真源命名和版本。
- 分镜统筹：只做单集概要、场次概要、分镜设计。当前阶段先不强制衔接资产审阅；正式生产时再接资产真源。
- 视频生成：只做视频提示词、资产包、视频候选和人工检查。

这样避免人类在一个一级页面里做一半再跳到另一个页面，符合连续生产习惯。
