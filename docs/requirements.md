# 需求方案

## 目标

构建一个面向长剧本短剧生产的 Web 系统。用户输入全集剧本后，系统辅助完成：

- 剧本格式清洗与审校
- 资产提取与资产描述
- 资产生图与版本锁定
- 多层上下文包生成
- 分镜脚本生成与复核
- 视频提示词生成
- 多格式交付导出

最终目标只有两类产物：

- 资产描述/资产生图提示词
- 分镜规划/视频提示词

其他产物都是中间产物，必须可靠、有效、精准，并且能被后续阶段明确使用。

## 核心约束

- 全剧剧本可达 50000 字或更长。
- 剧本包含多集、多场、多角色、多道具。
- 分镜不能简单逐段翻译剧本，必须依赖全剧、单集、场景、角色、道具、镜头连续状态。
- 系统必须适应不同题材、导演风格、输出格式、规则包和技能包。
- 所有关键产物必须可审阅、可编辑、可追踪、可导出。

## 需求编号

### R-001 剧本源头确认

系统必须支持剧本清洗稿，并让用户直观看到 LLM 后续使用的信息源头。

包括：

- 原始剧本输入
- 格式清洗稿
- 疑似错字/格式混乱/对白异常/角色名混用提示
- 清洗稿确认后进入后续流程

### R-002 长文本处理

系统必须能处理 50000 字级别全集剧本。

包括：

- 自动分集
- 自动分场
- 分块处理
- 上下文摘要
- 中间产物缓存
- 局部重跑
- 失败重试

### R-003 资产真源

系统必须生成角色、场景、道具资产真源。

资产必须支持：

- 统一命名
- 描述
- 连续性规则
- 生图提示词
- 候选图
- 锁定图
- 版本记录

### R-004 资产生图工作台

资产描述后续会用于模型生图，因此需要单独页面管理。

包括：

- 资产列表
- 生图提示词
- 生成结果
- 多版本选择
- 锁定版本
- 锁定图进入分镜提示词上下文

### R-005 上下文包

系统必须生成必要辅助信息，供分镜和提示词使用。

辅助信息必须遵守最小有效原则：下游明确使用、来源可追踪、可靠性可判断。

包括：

- 03 集级辅助：揭露顺序、情绪弧线、跨场关系、跨场道具约束、视觉策略
- 04 场级连续性：人物进出、空间关系、道具流转、场内时序
- 角色/场景/道具锁定项
- 镜头级连续性监看

### R-006 分镜生成

分镜生成必须是视听语言设计，不是剧本文字逐段翻译。

分镜生成输入必须包含：

- 当前剧本文字
- 03 集级辅助
- 04 场级连续性
- 锁定资产
- 场内空间时序
- 相邻镜头状态
- 风格规则

首次生成以场级为主，局部不满意时允许块级重跑。

### R-007 风格与规则包

系统必须支持不同题材、导演风格、规则包、技能包。

包括：

- 题材 profile
- 导演 profile
- 剧本清洗规则包
- 资产规则包
- 分镜规则包
- 生图提示词规则包
- 视频提示词规则包
- 校验规则包

### R-008 输出适配

系统必须支持多种输出格式。

当前默认：

- `assets.json`
- `shots.json`
- `prompts.json`
- `grouped_prompts.csv`
- 项目 ZIP

后续通过 Output Adapter 扩展。

### R-009 可追踪与可溯源

所有需求、设计决策、代码模块和实现状态必须有追踪关系。

包括：

- 需求编号
- 设计决策编号
- 代码模块
- 实现状态
- 后续风险

### R-010 项目管理与保存

系统必须支持多个项目、保存和版本。

P0 阶段先支持：

- 本地项目列表
- 新建项目
- 保存版本摘要
- 当前项目状态恢复

后续支持：

- 后端持久化
- 完整版本 diff
- 回滚
- 权限

### R-011 Pipeline 执行追踪

系统必须记录每次管线运行。

包括：

- `PipelineRun`
- `StageResult`
- 阶段输入/输出引用
- 阶段状态
- 阻塞原因
- 执行日志

### R-012 执行反馈与开发排查

系统底部必须有执行反馈区。

- 默认收起为窄状态栏
- 展开后为大排查面板
- 支持搜索和问题过滤
- 后续接 LLM 调用日志、任务队列日志、组件函数日志
